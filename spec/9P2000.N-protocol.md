# 9P2000.N Protocol Specification

## 1. Overview

9P2000.N (Next/Negotiated) is a modular extension to the 9P remote
resource protocol. It preserves full backward compatibility with 9P2000,
9P2000.u while adding 43 new message type pairs organized
into 8 independently negotiable capability domains.

### Design Principles

1. **Wire format preserved**: `size[4] type[1] tag[2] payload` — no header changes
2. **Composable negotiation**: Capabilities are independently negotiable via `Tcaps/Rcaps`
3. **Additive only**: No existing message semantics are modified
4. **No downgrade**: 9P2000.N requires both sides to support the protocol; no fallback to older versions
5. **Server-push via P9_NOTAG**: Three message types use `tag = 0xFFFF` for unsolicited server-to-client messages

### Data Encoding

All integers are little-endian. Strings are encoded as `length[2]` followed by UTF-8 bytes (no NUL terminator). QIDs are 13 bytes: `type[1] version[4] path[8]`.

## 2. Capability Negotiation Framework

### Version Handshake

The client proposes `"9P2000.N"` in `Tversion`. If the server supports it, it responds `"9P2000.N"` in `Rversion`. Otherwise, the server responds with `"unknown"` and the connection MUST be closed — 9P2000.N does not fall back to older protocol versions.

### Tcaps / Rcaps (type 128/129)

Immediately after receiving `Rversion` confirming `"9P2000.N"`, the client sends `Tcaps` listing the capabilities it desires. The server responds with the subset it supports.

```
Tcaps: size[4] 128 tag[2] ncaps[2] ncaps*(cap[s])
Rcaps: size[4] 129 tag[2] ncaps[2] ncaps*(cap[s])
```

Capability strings use hierarchical naming: `domain.feature[.version]`

**Defined domains:**

| Domain | Description |
|--------|-------------|
| `security.*` | Security and authentication |
| `perf.*` | Performance and efficiency |
| `fs.*` | Modern filesystem semantics |
| `dist.*` | Distributed systems support |
| `obs.*` | Observability and monitoring |
| `res.*` | Resource management |
| `stream.*` | Streaming and async operations |
| `content.*` | Content-awareness |
| `transport.*` | Transport-layer negotiation (QUIC, RDMA) |

## 3. Security Extensions (`security.*`)

### 3.1 Tstartls / Rstartls (type 130/131) — `security.tls`

```
Tstartls: size[4] 130 tag[2]
Rstartls: size[4] 131 tag[2]
```

After `Rstartls`, both sides begin a TLS handshake on the underlying transport. All subsequent 9P messages are inside the TLS channel. If the handshake fails, the connection is torn down.

### 3.2 Tauthneg / Rauthneg (type 132/133) — `security.auth`

```
Tauthneg: size[4] 132 tag[2] nmechs[2] nmechs*(mech[s])
Rauthneg: size[4] 133 tag[2] mech[s] challenge_len[4] challenge[challenge_len]
```

Client proposes authentication mechanisms. Server selects one and optionally includes an initial challenge. Subsequent auth data flows through the existing `Tauth/Rauth` fid-based channel.

**Supported mechanisms:**

| Mechanism | Description |
|-----------|-------------|
| `SASL-SCRAM-SHA-256` | SCRAM password-based authentication |
| `mTLS` | Mutual TLS certificate authentication |
| `OIDC` | OpenID Connect token authentication |
| `P9any` | Plan 9 legacy authentication |
| `SPIFFE-X.509` | SPIFFE X.509-SVID mutual TLS |
| `SPIFFE-JWT` | SPIFFE JWT-SVID bearer token |

### 3.3 Tcapgrant / Rcapgrant (type 134/135) — `security.caps`

```
Tcapgrant: size[4] 134 tag[2] fid[4] rights[8] expiry[8] depth[2]
Rcapgrant: size[4] 135 tag[2] token[s]
```

- `rights[8]`: bitmask (read=0x1, write=0x2, walk=0x4, create=0x8, remove=0x10, setattr=0x20, lock=0x40, admin=0x80)
- `expiry[8]`: nanosecond Unix timestamp (0 = no expiry)
- `depth[2]`: walk levels below fid (0 = this fid only, 0xFFFF = unlimited)
- `token[s]`: opaque server-generated bearer token

### 3.4 Tcapuse / Rcapuse (type 136/137) — `security.caps`

```
Tcapuse: size[4] 136 tag[2] fid[4] token[s]
Rcapuse: size[4] 137 tag[2] qid[13]
```

Attach a fid to a subtree using a capability token instead of normal auth.

### 3.5 Tauditctl / Rauditctl (type 138/139) — `security.audit`

```
Tauditctl: size[4] 138 tag[2] fid[4] flags[4]
Rauditctl: size[4] 139 tag[2]
```

- `flags[4]`: operations to audit (read=0x1, write=0x2, walk=0x4, create=0x8, remove=0x10, setattr=0x20, all=0xFFFFFFFF)

### 3.6 SPIFFE Identity Integration (`security.spiffe`)

SPIFFE (Secure Production Identity Framework for Everyone) integration enables
workload-to-workload identity verification using X.509-SVIDs and JWT-SVIDs.
When `security.spiffe` is negotiated, three new message pairs become available.

#### 3.6.1 Tstartls_spiffe / Rstartls_spiffe (type 140/141)

```
Tstartls_spiffe: size[4] 140 tag[2] spiffe_id[s] trust_domain[s]
Rstartls_spiffe: size[4] 141 tag[2] spiffe_id[s] trust_domain[s]
```

SPIFFE-aware TLS handshake. Unlike plain `Tstartls`, both sides exchange their
SPIFFE ID and trust domain *before* the TLS handshake begins. After
`Rstartls_spiffe`, both sides perform mutual TLS using X.509-SVIDs. The TLS
certificates MUST contain the SPIFFE X.509-SVID extension (OID
`1.3.6.1.4.1.51483.2.1`), and the SPIFFE ID in the certificate MUST match
the `spiffe_id` field declared in the message.

- `spiffe_id[s]`: Full SPIFFE ID URI (e.g., `"spiffe://example.com/server"`)
- `trust_domain[s]`: Trust domain portion (e.g., `"example.com"`)

#### 3.6.2 Tfetchbundle / Rfetchbundle (type 142/143)

```
Tfetchbundle: size[4] 142 tag[2] trust_domain[s] format[1]
Rfetchbundle: size[4] 143 tag[2] trust_domain[s] format[1] bundle_len[4] bundle[bundle_len]
```

Fetch the SPIFFE trust bundle for a given trust domain. This is essential for
verifying SVIDs from remote trust domains (federation).

- `format[1]`: 0=X.509 CAs (PEM-encoded CA certificate chain), 1=JWT JWK Set (JSON)
- `bundle[bundle_len]`: The trust bundle data in the requested format
- Clients SHOULD cache bundles and refetch when SVID verification fails

For X.509 format, the bundle contains the root CA certificates that sign SVIDs
in the specified trust domain. For JWT format, the bundle contains the JWK Set
with public keys used to sign JWT-SVIDs.

#### 3.6.3 Tspiffeverify / Rspiffeverify (type 144/145)

```
Tspiffeverify: size[4] 144 tag[2] svid_type[1] spiffe_id[s] svid_len[4] svid[svid_len]
Rspiffeverify: size[4] 145 tag[2] status[1] spiffe_id[s] expiry[8]
```

Explicit SVID verification request. The client presents its SVID (either X.509
certificate chain or JWT token) and the server verifies it against its trust
bundle store.

- `svid_type[1]`: 0=X.509 (DER-encoded certificate chain), 1=JWT (token string)
- `status[1]`: 0=OK, 1=untrusted trust domain, 2=expired, 3=revoked, 4=ID mismatch
- `expiry[8]`: SVID expiry timestamp (nanosecond Unix timestamp)

#### SPIFFE Connection Flow

```
Client                                  Server
  |                                       |
  |--- Tversion "9P2000.N" msize -------->|
  |<-- Rversion "9P2000.N" msize ---------|
  |                                       |
  |--- Tcaps [security.spiffe, ...] ----->|
  |<-- Rcaps [security.spiffe, ...] ------|
  |                                       |
  |--- Tfetchbundle "example.com" 0 ----->|  (fetch server's X.509 trust bundle)
  |<-- Rfetchbundle "example.com" 0 PEM --|
  |                                       |
  |--- Tstartls_spiffe ------------------>|  (exchange SPIFFE IDs)
  |     spiffe_id="spiffe://a.com/client" |
  |     trust_domain="a.com"              |
  |<-- Rstartls_spiffe -------------------|
  |     spiffe_id="spiffe://b.com/server" |
  |     trust_domain="b.com"              |
  |        <<< mTLS with X.509-SVIDs >>>  |
  |                                       |
  |--- Tauthneg [SPIFFE-X.509] ---------->|
  |<-- Rauthneg "SPIFFE-X.509" -----------|
  |--- Tauth (SPIFFE auth exchange) ----->|
  |<-- Rauth -----------------------------|
  |                                       |
  |--- Tattach fid afid uname aname ----->|
  |<-- Rattach qid -----------------------|
```

#### JWT-SVID as Capability Token Convention

When `security.spiffe` and `security.caps` are both negotiated, the `token[s]`
field in `Rcapgrant` MAY contain a JWT-SVID. The JWT-SVID MUST include:

| JWT Claim | Mapping |
|-----------|---------|
| `sub` | SPIFFE ID (e.g., `spiffe://example.com/workload`) |
| `exp` | Expiry (maps to `Rcapgrant.expiry`) |
| `aud` | Server SPIFFE ID (audience restriction) |
| `p9n_rights` | Custom claim: rights bitmask as hex string |
| `p9n_depth` | Custom claim: walk depth limit |

Servers that issue JWT-SVIDs as capability tokens MUST sign them with keys
present in the trust bundle fetchable via `Tfetchbundle`.

## 4. Performance Extensions (`perf.*`)

### 4.1 Tcompound / Rcompound (type 156/157) — `perf.compound`

```
Tcompound: size[4] 156 tag[2] nops[2] nops*(opsize[4] type[1] payload[...])
Rcompound: size[4] 157 tag[2] nresults[2] nresults*(opsize[4] type[1] payload[...])
```

Sub-operations execute sequentially. If any returns an error, subsequent operations are skipped and the result array is truncated at the error.

**Magic fid**: `0xFFFFFFFE` in any sub-op's payload refers to the fid returned by the most recent sub-op that created/walked a fid. This enables `walk+open+read+clunk` in a single round-trip without knowing the fid in advance.

Sub-op format: `opsize[4]` includes itself and `type[1]` (total 5 bytes minimum). The `type[1]` is any valid 9P or 9P2000.N message type. There is no `tag` field in sub-ops.

### 4.2 `perf.largemsg` (capability flag only)

No new messages. When negotiated, `msize` in `Tversion/Rversion` is allowed up to 2^31−1 (2 GB).

### 4.3 Tcompress / Rcompress (type 158/159) — `perf.compress`

```
Tcompress: size[4] 158 tag[2] algo[1] level[1]
Rcompress: size[4] 159 tag[2] algo[1]
```

- `algo[1]`: 0=none, 1=lz4, 2=zstd, 3=snappy
- After `Rcompress`, all subsequent message payloads are compressed; headers remain uncompressed
- `size[4]` reflects wire size (compressed)
- Send `algo=0` to disable compression

### 4.4 `perf.zerocopy` (capability flag only)

No new messages. When negotiated, read/write payloads must be page-aligned and `iounit` is guaranteed to be a multiple of the system page size.

### 4.5 Tcopyrange / Rcopyrange (type 160/161) — `perf.copy`

```
Tcopyrange: size[4] 160 tag[2] srcfid[4] srcoff[8] dstfid[4] dstoff[8] count[8] flags[4]
Rcopyrange: size[4] 161 tag[2] count[8]
```

- `flags[4]`: reflink=0x1 (CoW), dedup=0x2
- Maps to `copy_file_range(2)` / `FICLONE`

### 4.6 Tallocate / Rallocate (type 162/163) — `perf.alloc`

```
Tallocate: size[4] 162 tag[2] fid[4] mode[4] offset[8] length[8]
Rallocate: size[4] 163 tag[2]
```

- `mode[4]`: matches Linux `fallocate` flags (KEEP_SIZE=0x1, PUNCH_HOLE=0x2, COLLAPSE_RANGE=0x4, ZERO_RANGE=0x8, INSERT_RANGE=0x10)

### 4.7 Tseekhole / Rseekhole (type 164/165) — `perf.alloc`

```
Tseekhole: size[4] 164 tag[2] fid[4] type[1] offset[8]
Rseekhole: size[4] 165 tag[2] offset[8]
```

- `type[1]`: 0=SEEK_DATA, 1=SEEK_HOLE

### 4.8 Tmmaphint / Rmmaphint (type 166/167) — `perf.mmap`

```
Tmmaphint: size[4] 166 tag[2] fid[4] offset[8] length[8] prot[4]
Rmmaphint: size[4] 167 tag[2] granted[1]
```

Hint to the server to optimize caching for random page-level access.

## 11. Transport Extensions (`transport.*`)

Transport extensions allow 9P2000.N to negotiate transport-specific optimizations
without changing the core protocol semantics. When a `transport.*` capability is
negotiated, certain protocol behaviors are adjusted to eliminate redundancy with
the underlying transport or to exploit hardware capabilities.

### 11.1 Transport Capabilities (flag only, no messages)

The following capabilities are negotiated via `Tcaps/Rcaps` but do not define
new message types. They modify the behavior of existing messages:

- **`transport.quic`**: Underlying transport is QUIC. Implications:
  - `Tstartls/Rstartls` MUST NOT be sent (QUIC has built-in TLS 1.3)
  - `Tstartls_spiffe` is simplified to identity exchange only (TLS certs already
    verified during QUIC handshake)
  - `Tsession` is used only for 9P state recovery (fid/lease/watch); transport-layer
    session resumption is handled by QUIC 0-RTT
  - Connection migration is provided by QUIC transparently

- **`transport.quic.multistream`**: When negotiated alongside `transport.quic`,
  enables mapping of individual 9P operations to separate QUIC streams:
  - Stream 0: control channel (Tversion, Tcaps, Tsession)
  - Stream 2*tag: request/response for tag N
  - Server-initiated unidirectional streams: server-push messages (replaces P9_NOTAG)
  - Eliminates head-of-line blocking between concurrent operations
  - Tag space no longer limits concurrency (QUIC stream ID space is 2^62)

- **`transport.cxl`**: Underlying transport includes CXL.mem shared memory.
  Implications:
  - Files mapped via `Tcxlmap` are accessed through CPU load/store instructions
    instead of `Tread`/`Twrite` messages — data plane is fully separated from
    control plane
  - `Tmmaphint` is superseded for CXL-mapped regions (DAX-style direct access)
  - When CXL.cache is available between hosts, `Tlease` for data coherence is
    optional; the server may report `coherence=hardware` in `Rcxlmap` indicating
    that hardware snooping maintains data consistency
  - `Tlease` remains required for **metadata** coherence (stat, dentry, xattr)
    since metadata structures are managed by the server, not shared memory
  - Control messages (walk, stat, create, watch, etc.) continue to use the
    underlying control channel (CXL.io MMIO doorbell or fallback transport)

- **`transport.rdma`**: Underlying transport is RDMA. Implies:
  - Control messages (Tcaps, Twatch, Tlease, etc.) use Send/Recv verbs
  - Data-heavy messages may reference pre-registered memory regions via `rdma_token`
  - `perf.zerocopy` is implicitly active

### 11.2 Trdmatoken / Rrdmatoken (type 150/151) — `transport.rdma`

```
Trdmatoken: size[4] 150 tag[2] fid[4] direction[1] rkey[4] addr[8] length[4]
Rrdmatoken: size[4] 151 tag[2] rkey[4] addr[8] length[4]
```

Exchange RDMA memory region tokens. The client registers a memory region and sends
its remote key (`rkey`) and address to the server, or vice versa. Once exchanged,
`Tread`/`Twrite` on the associated fid can use RDMA Read/Write verbs for zero-copy
data transfer instead of inline message payloads.

- `direction[1]`: 0=client-to-server (client sends buffer for server to RDMA Write into),
  1=server-to-client (server sends buffer for client to RDMA Read from)
- `rkey[4]`: RDMA remote key for memory access
- `addr[8]`: virtual address of the registered memory region
- `length[4]`: size of the registered region in bytes

After token exchange, data operations on this fid work as follows:

**Read path** (direction=0, client provides buffer):
1. Client sends `Tread fid offset count` via Send/Recv (control channel)
2. Server executes RDMA Write to client's registered memory
3. Server sends `Rread count` (header only, no inline data) via Send/Recv

**Write path** (direction=0, client provides buffer):
1. Client places data in registered memory
2. Client sends `Twrite fid offset count` via Send/Recv (signals data is ready)
3. Server executes RDMA Read from client's memory
4. Server sends `Rwrite count` via Send/Recv

### 11.3 Trdmanotify / Rrdmanotify (type 152/153) — `transport.rdma`

```
Trdmanotify: size[4] 152 tag[2] rkey[4] addr[8] length[4] slots[2]
Rrdmanotify: size[4] 153 tag[2]
```

Register a notification ring buffer for RDMA-based server push. The client
allocates a ring buffer, registers it as an RDMA memory region, and tells the
server to use RDMA Write with Immediate Data for push messages.

- `rkey[4]`, `addr[8]`, `length[4]`: client's ring buffer memory region
- `slots[2]`: number of fixed-size slots in the ring buffer

Server pushes (Rnotify, Rleasebreak, Rstreamdata) are delivered via:
1. Server RDMA Writes the message payload into the next ring buffer slot
2. Server sets Immediate Data = `(slot_index << 16) | msg_type`
3. Client's Completion Queue receives the immediate data notification
4. Client reads the message from the corresponding slot — zero syscalls, zero copies

### 11.4 Tquicstream / Rquicstream (type 154/155) — `transport.quic.multistream`

```
Tquicstream: size[4] 154 tag[2] stream_type[1] stream_id[8]
Rquicstream: size[4] 155 tag[2] stream_id[8]
```

Bind a logical 9P channel to a specific QUIC stream. Used to set up dedicated
streams for different purposes:

- `stream_type[1]`:
  - 0 = control (version/caps/session negotiation)
  - 1 = data (normal request/response operations)
  - 2 = push (server-initiated notifications, unidirectional)
  - 3 = bulk (streaming read/write, maps Tstreamdata to QUIC stream flow control)

After binding, messages of the given type are sent on the specified QUIC stream.
This replaces the `P9_NOTAG` mechanism for server push when `transport.quic.multistream`
is active — push messages arrive on their dedicated unidirectional stream with
normal tags, eliminating the need for the special tag sentinel.

#### `stream_id` semantics

The interpretation of the `stream_id[8]` field in the request depends on
`stream_type`, because different channels have different rules about which
side opens the underlying QUIC stream:

- **`stream_type == 2` (push)**: the server is the initiator of the
  unidirectional QUIC stream. The client has no way to hold a valid stream
  identifier at request time, so the request `stream_id` field is reserved.
  Clients MUST send `stream_id = 0`; servers MUST return `Rlerror(EINVAL)`
  for any non-zero value.

- **`stream_type ∈ {0, 1, 3}`**: the semantics of the request `stream_id`
  field are reserved for a future revision of this specification. The field
  may carry a client-provided QUIC stream identifier that the server binds
  the channel to — for example, a bidirectional stream the client has
  already opened. Until such a revision exists, servers MUST NOT attempt to
  interpret the field for these stream_types, and MAY return
  `Rlerror(EOPNOTSUPP)` for the entire request if the stream_type itself is
  not implemented.

The response `stream_id[8]` is an opaque, server-assigned alias for the
bound stream. For `stream_type == 2`, the server derives it from the
underlying transport-level identifier of the uni-stream it opened (for
example, the QUIC stream ID cast to `u64`). The client SHOULD treat the
returned value as an opaque handle — it is informational only and is not
required for routing incoming push messages, which the client demultiplexes
by the "message arrived on a unidirectional stream" property regardless of
the stream's id.

#### Evolution of request `stream_id`

Future revisions that assign meaning to the request `stream_id` field for
other stream_types MUST advertise the new behavior via a distinct capability
bit (for example, `transport.quic.multistream.v2`). A server that does not
advertise the new capability continues to enforce the rules above: `MUST
be 0` for `stream_type == 2`, and `EOPNOTSUPP` for unimplemented
stream_types. This ensures a current-spec server cannot misinterpret a
request from a future-spec client.

#### Error conditions

| Condition | Response |
|---|---|
| `stream_type == 2` and `stream_id != 0` | `Rlerror(EINVAL)` |
| `stream_type` is unimplemented on this server | `Rlerror(EOPNOTSUPP)` |
| `transport.quic.multistream` not negotiated | `Rlerror(EOPNOTSUPP)` |
| The client is not connected over QUIC (e.g. TCP+TLS) | `Rlerror(EOPNOTSUPP)` |
| A binding for this `stream_type` already exists on this session | `Rlerror(EBUSY)` |

### 11.5 Tcxlmap / Rcxlmap (type 146/147) — `transport.cxl`

```
Tcxlmap: size[4] 146 tag[2] fid[4] offset[8] length[8] prot[4] flags[4]
Rcxlmap: size[4] 147 tag[2] hpa[8] length[8] granularity[4] coherence[1]
```

Map a file region into CXL shared memory. After a successful `Rcxlmap`, the
client configures its CXL HDM (Host-managed Device Memory) decoder to access
the returned Host Physical Address (HPA) range, and can then use CPU load/store
instructions for data access — no `Tread`/`Twrite` messages required.

Request fields:
- `fid[4]`: open file to map
- `offset[8]`: file offset to start mapping
- `length[8]`: requested mapping length (0 = whole file)
- `prot[4]`: access protection (PROT_READ=0x1, PROT_WRITE=0x2)
- `flags[4]`: MAP_SHARED=0x1, MAP_PRIVATE=0x2, MAP_DAX=0x4

Response fields:
- `hpa[8]`: Host Physical Address on the CXL device
- `length[8]`: actual mapped length (may be less than requested)
- `granularity[4]`: CXL interleave granularity in bytes (typically 256 or 4096)
- `coherence[1]`: coherence mode in effect for this mapping:
  - 0 = `SOFTWARE` — no CXL.cache between hosts; client MUST hold a `Tlease`
    for safe caching and will receive `Rleasebreak` on conflicts
  - 1 = `HARDWARE` — CXL.cache snooping active; hardware maintains data
    coherence across all hosts with active mappings; `Tlease` is not needed
    for data but still required for metadata
  - 2 = `HYBRID` — hardware coherence within the same CXL fabric; software
    lease across fabrics or for remote clients without CXL

Data access after mapping:
1. Client maps HPA into its virtual address space via the CXL HDM decoder
2. Reads: CPU `load` instructions (ns latency, hardware-coherent)
3. Writes: CPU `store` instructions (with `clflush`/`clwb` for persistence)
4. `Tread`/`Twrite` for this region are no longer needed
5. `Tfsync` still required to ensure persistence on the CXL device

Unmapping: `Tclunk` on the fid invalidates the mapping. Client MUST stop
accessing the HPA range before clunking.

### 11.6 Tcxlcoherence / Rcxlcoherence (type 148/149) — `transport.cxl`

```
Tcxlcoherence: size[4] 148 tag[2] fid[4] mode[1]
Rcxlcoherence: size[4] 149 tag[2] mode[1] snoop_id[4]
```

Negotiate the coherence mode for a CXL-mapped file. This allows the client to
request a specific coherence strategy, and the server responds with the mode
actually in effect.

- `mode[1]` (request):
  - 0 = `SOFTWARE` — use 9P Tlease only (fallback for mixed CXL/non-CXL clients)
  - 1 = `HARDWARE` — use CXL.cache snooping only (lowest latency, requires all
    clients to be on the same CXL fabric)
  - 2 = `HYBRID` — hardware coherence when possible, software lease as fallback
  - 3 = `RELAXED` — allow stale reads (no coherence guarantees, highest throughput)

- `mode[1]` (response): the mode the server actually granted (may differ from request)
- `snoop_id[4]`: CXL snoop filter identifier; clients use this to configure their
  local CXL.cache agent for the correct snoop domain (0 = not applicable)

Coherence behavior by mode:

| Mode | Data reads | Data writes | Metadata | Lease needed? |
|------|-----------|-------------|----------|---------------|
| SOFTWARE | via Tread or cached with lease | via Twrite or cached with write lease | Tlease | Yes (data + metadata) |
| HARDWARE | load (hw coherent) | store (hw coherent) | Tlease | Metadata only |
| HYBRID | load (hw) or Tread (remote) | store (hw) or Twrite (remote) | Tlease | Data: depends on path; Metadata: always |
| RELAXED | load (may be stale) | store (eventual) | Twatch for notifications | No |

## 12. Server-Push Protocol

Three message types are server-initiated (unsolicited):

| Type | Name | Tag | Trigger |
|------|------|-----|---------|
| 185 | Rnotify | 0xFFFF | Watched file/directory changes |
| 205 | Rleasebreak | 0xFFFF | Server needs to revoke/downgrade lease |
| 247 | Rstreamdata | 0xFFFF | Stream read data delivery |

Clients that negotiate push-capable extensions must be prepared to receive messages with `tag = P9_NOTAG` at any time. The `watchid`, `leaseid`, or `streamid` field within the message identifies the context.

## 13. Connection Lifecycle

### 13.1 Full 9P2000.N Handshake

```
Client                                  Server
  |                                       |
  |--- Tversion "9P2000.N" msize -------->|
  |<-- Rversion "9P2000.N" msize ---------|
  |                                       |
  |--- Tcaps [security.tls, ...] -------->|
  |<-- Rcaps [security.tls, ...] ---------|
  |                                       |
  |--- Tstartls ------------------------->|  (if security.tls negotiated)
  |<-- Rstartls --------------------------|
  |        <<< TLS handshake >>>          |
  |                                       |
  |--- Tauthneg [SCRAM-SHA-256, ...] ---->|  (if security.auth negotiated)
  |<-- Rauthneg "SCRAM-SHA-256" chal -----|
  |--- Tauth afid uname aname ----------->|
  |<-- Rauth aqid ------------------------|
  |        <<< auth exchange >>>          |
  |                                       |
  |--- Tattach fid afid uname aname ----->|
  |<-- Rattach qid -----------------------|
  |                                       |
  |--- Tsession key flags --------------->|  (if dist.session negotiated)
  |<-- Rsession flags --------------------|
  |                                       |
  |    <<< normal 9P operations >>>       |
  |    <<< + server push messages >>>     |
```

### 13.2 No Backward Compatibility

9P2000.N does NOT fall back to 9P2000.L or earlier protocol versions. If the
server responds with a version string other than `"9P2000.N"` in `Rversion`,
the client MUST close the connection.

### 13.3 Base Message Format: Tattach

9P2000.N uses the standard 9P2000 `Tattach` message format (without the
`n_uname` field added by 9P2000.L). User identity is established through
the `Tauthneg` / SPIFFE authentication flow, not through numeric UID
assertion:

```
Tattach: size[4] 104 tag[2] fid[4] afid[4] uname[s] aname[s]
Rattach: size[4] 105 tag[2] qid[13]
```

The `uname` field is a human-readable hint only. The server MUST derive
the authenticated identity from the preceding `Tauthneg` exchange or
transport-layer credentials, not from `uname`.

## Appendix A: Transport Profiles

9P2000.N supports multiple transport layers. Each transport changes which
protocol features are active, which are redundant, and how data flows between
client and server. This appendix provides a complete comparison.

### A.1 Transport Overview

```
              9P2000.N Protocol Layer
                      │
       ┌──────────────┼──────────────────────┐
       │         Transport Interface         │
       │   connect / send / recv / close     │
       │   + transport-specific extensions   │
       └──────────────┼──────────────────────┘
                      │
   ┌──────┬───────────┼──────────┬───────────┐
   │      │           │          │           │
 TCP+TLS  QUIC      virtio     RDMA        CXL
   │      │           │          │           │
 kernel  userspace  hypervisor  NIC hw     memory
 stack   UDP+TLS    (PCI)      bypass     semantic
```

| Property | TCP+TLS | QUIC | RDMA | CXL |
|----------|---------|------|------|-----|
| **Typical latency** | ~1 ms | ~1 ms | ~1-5 μs | ~100-300 ns |
| **CPU overhead** | high (kernel + TLS) | medium (userspace) | very low (hw offload) | near zero (load/store) |
| **Data path** | kernel copy | userspace copy | NIC DMA (zero-copy) | CPU direct (no transfer) |
| **Encryption** | TLS (explicit) | TLS 1.3 (built-in) | none (needs IPsec) | none (physical) |
| **Multi-host** | yes (WAN) | yes (WAN+mobile) | datacenter only | same CXL fabric |
| **Cache coherence** | software (lease) | software (lease) | software (lease) | hardware (CXL.cache) |
| **Connection migration** | no | yes | no | N/A |
| **Zero-copy data** | no | no | yes (RDMA verbs) | yes (load/store) |
| **NAT/firewall** | TCP friendly | UDP, good traversal | no NAT | N/A (physical) |
| **Hardware required** | any NIC | any NIC | RDMA NIC | CXL device + switch |

### A.2 Protocol Behavior Differences

The following table shows how each transport affects the behavior of 9P2000.N
protocol features. "**skip**" means the feature is redundant and MUST NOT be
used; "**replaced**" means the feature is superseded by a transport-native
mechanism; "**unchanged**" means the feature works identically to TCP.

| Protocol Feature | TCP+TLS | QUIC | RDMA | CXL |
|-----------------|---------|------|------|-----|
| **Tversion/Rversion** | unchanged | unchanged | unchanged | unchanged |
| **Tcaps/Rcaps** | unchanged | unchanged | unchanged | unchanged |
| **Tstartls** | required for encryption | **skip** (TLS built-in) | unchanged | unchanged |
| **Tstartls_spiffe** | full (ID + TLS) | ID exchange only (TLS in QUIC) | full | full |
| **Tsession** | full (reconnect) | 9P state only (QUIC handles transport resume) | full | full |
| **Tcompound** | critical (saves RTTs) | useful (saves stream setup) | useful (saves doorbells) | metadata only (data is load/store) |
| **Tread/Twrite** (data) | inline in message | inline in message | via RDMA Read/Write verbs | **replaced** by CXL load/store |
| **Tread/Twrite** (metadata) | unchanged | unchanged | via Send/Recv | unchanged (control channel) |
| **tag multiplexing** | required (65534 max) | optional if multistream (2^62 streams) | required | required |
| **Server push (P9_NOTAG)** | on same connection | dedicated QUIC stream (if multistream) | via RDMA Write+Imm to ring buffer | via CXL Write+doorbell |
| **Tlease** (data) | required for caching | required for caching | required for caching | **optional** if CXL.cache available |
| **Tlease** (metadata) | required | required | required | **required** (always) |
| **Tcompress** | full payload compression | full payload compression | control channel only (data is zero-copy) | control channel only |
| **Tcopyrange** | saves 2× network transfer | saves 2× network transfer | saves RDMA verbs | N/A (both files in shared memory) |
| **Tfsync** | unchanged | unchanged | unchanged | required (CXL persistence) |
| **Twatch/Rnotify** | unchanged | unchanged (or via push stream) | via RDMA notification ring | via CXL doorbell + shared ring |

### A.4 Feature Matrix: What to Use When

| Scenario | Recommended Transport | Key Advantage |
|----------|----------------------|---------------|
| Edge node mounting cloud storage | QUIC | NAT traversal, 0-RTT resume, connection migration |
| Cross-datacenter file replication | TCP+TLS or QUIC | Reliable over WAN, encryption |
| VM host file sharing (QEMU) | virtio (existing) | Mature, low overhead |

```
                   ┌───────────────┐
  WAN client ──────┤ TCP+TLS       │
  (laptop)         │               │
                   │  Metadata +   │
  Mobile app ──────┤ QUIC          │──── Storage
  (phone)          │  Data inline  │     Backend
                   │               │
  HPC node ────────┤ RDMA          │
  (same rack)      │  Data: verbs  │
                   │               │
  GPU server ──────┤ CXL           │
  (same fabric)    │  Data: memory │
                   └───────────────┘

Each client negotiates its own capability set:
  - TCP client:  [security.tls, perf.compound, dist.lease]
  - QUIC client: [transport.quic, perf.compound, dist.lease]
  - RDMA client: [transport.rdma, perf.compound, dist.lease]
  - CXL client:  [transport.cxl, dist.lease]  (no perf.compound for data)
```

### A.6 Coherence Across Transports

When clients on different transports access the same file, coherence is
managed through the **most conservative** mechanism available to all
participants:

| Client A transport | Client B transport | Data coherence mechanism |
|-------------------|-------------------|------------------------|
| TCP | TCP | 9P Tlease (software) |
| TCP | RDMA | 9P Tlease (software) |
| TCP | CXL | 9P Tlease (software — TCP client cannot use CXL.cache) |
| RDMA | RDMA | 9P Tlease (software) |
| RDMA | CXL | 9P Tlease (software — RDMA client has no CXL.cache) |
| CXL | CXL (same fabric) | CXL.cache (hardware — lowest latency) |
| CXL | CXL (different fabric) | 9P Tlease (software — cross-fabric) |

The server tracks which coherence mechanism each client supports and issues
`Rleasebreak` only to clients that hold software leases. CXL clients on the
same fabric are transparent to each other via hardware snooping.
